# Hive — Workflow Reference

## Bee Terminology (Agent Roles)

| Role | Who | Responsibility |
|------|-----|----------------|
| Beekeeper | You (human) | Observer, approver, decision-maker |
| Hive Bee (hive-master) | Hybrid planner+orchestrator | Plans AND orchestrates (unified mode) |
| Architect Bee | Planner only | NEVER executes or delegates code — plan only |
| Swarm Bee | Orchestrator | Coordinates execution, handles blocked workers |
| Scout Bee | Researcher | MCP tools: grep_app, context7, websearch |
| Forager Bee | Implementer | Codes in isolated worktrees, best-effort verification |
| Hygienic Bee | Reviewer | Returns OKAY/REJECT verdict |

**agentMode** (in config):
- `unified` (default): hive-master handles both planning + orchestration
- `dedicated`: separate architect-planner + swarm-orchestrator agents

## .hive/ Directory Structure

```
.hive/features/<feature-name>/
├── feature.json                  # Feature metadata, status
├── plan.md                       # The approved plan
├── tasks.json                    # Task list (generated by hive_tasks_sync)
├── contexts/
│   └── architecture.md           # Royal Jelly — survives sessions (via hive_context_write)
└── tasks/
    ├── 01-extract-auth-logic/
    │   ├── status.json           # Task status + timestamps
    │   ├── spec.md               # Task context: prior/upcoming tasks, references
    │   └── report.md             # Summary, files changed, diff stats
    └── 02-setup-jwt-utils/
        └── ...
```

Context files (Royal Jelly) are the only persistent memory across sessions. Write decisions there — not in messages or notepads.

## Batched Parallelism (Principle 5)

NOT arbitrary parallelism. Tasks in same batch run in parallel; batches run sequentially.

```
Batch 1 (parallel):          Context flows forward
├── Task A                           ↓
├── Task B          → hive_context_write(batch-1-results)
└── Task C                           ↓
                         Batch 2 (parallel, context-aware):
                         ├── Task D (uses A+B+C results)
                         └── Task E (uses A+B+C results)
```

A **glue task** (or the orchestrator directly) synthesizes Batch N results before Batch N+1 begins. Without this, later tasks lack context.

## Hard Gates

- `hive_plan_write` requires `## Discovery` section ≥100 chars (regex-checked, hard rejection)
- `hive_worktree_commit` has a `verificationNote` advisory (not a hard rejection — best-effort is sufficient)

## Task Type Auto-inference

`buildSpecContent()` infers task type from name:
- "extract", "refactor" → refactoring
- "test", "spec" → testing
- "fix", "bug" → bugfix
- "setup", "scaffold" → greenfield
- default → modification

## Two-Tier Verification

| Tier | Who | What | Confidence |
|------|-----|------|-----------|
| Worker (forager) | forager-worker | ast-grep, no project deps | ~80% |
| Orchestrator | hive-master / swarm | Full build + test suite | High |

Workers do lightweight verification in isolated worktrees (no npm install etc.). Orchestrator runs full suite after each batch merge and re-dispatches if failures found.

## Blocked Worker Protocol (Detailed)

1. Worker detects it cannot proceed (missing info, conflicting requirement, ambiguous spec)
2. Worker: `hive_worktree_commit(status="blocked", blocker={reason, options, recommendation})`
   - `reason`: what's blocking
   - `options`: array of possible paths forward
   - `recommendation`: which option the worker suggests
3. Swarm asks user: **must use `question()` tool** (NOT plain text message)
4. Swarm resumes: `hive_worktree_create(task="...", continueFrom="blocked", decision="<user answer>")`
5. New forager spawns in the **same worktree** with the decision prepended to its context

Key: the new forager reads the existing worktree state — partial work is preserved.

---

## Worker Pre-flight (Orient Phase)

Before coding, every forager should:
1. Read `spec.md` (task context, prior task results)
2. Read relevant `contexts/` files (architecture decisions, Royal Jelly)
3. Check existing patterns in codebase
4. Verify assumptions match reality

---

## Post-batch Review (Optional)

After a batch merges, the Beekeeper can invoke hygienic-reviewer for a code review checkpoint. Returns OKAY or REJECT. If REJECT, orchestrator diagnoses and re-dispatches.

## Key Hive Tools

```
hive_feature_create(name)                          # Create feature
hive_plan_write(content, feature?)                 # Write plan.md
hive_plan_read(feature?)                           # Read plan + user comments
hive_plan_approve(feature?)                        # Approve plan
hive_tasks_sync(feature?)                          # Generate tasks from plan
hive_worktree_create(task, feature?, continueFrom?, decision?)  # Launch forager
hive_worktree_commit(task, summary, status, blocker?, feature?) # Complete/block task
hive_worktree_discard(task, feature?)              # Abort + reset task
hive_merge(task, strategy?, feature?)              # Merge branch into current
hive_context_write(name, content, feature?)        # Persist context
hive_status(feature?)                              # Full feature state
hive_task_create(name, feature?, order?)           # Manual task creation
hive_task_update(task, status?, summary?, feature?) # Update task state
```
