# Dokploy workflows reference

This reference supports the `dokploy-admin` skill with concrete guidance distilled from Dokploy docs, Dokploy MCP docs, Docker docs, and local operational notes.

## Resource types that matter most

- **Projects / environments** organize shared context and variables.
- **Applications** are single-service deployments with source/build/provider settings.
- **Databases** are first-class resources and usually require explicit deploy steps.
- **Docker Compose / Stack** support multi-service deployments.
- **Domains** control Traefik routing and public reachability.

## Prefer MCP for routine operations

Useful MCP patterns:

### Project and application setup

1. List or inspect projects
2. Create or inspect the application
3. Configure source provider
4. Configure build type
5. Save environment variables
6. Create domain
7. Deploy and inspect status

Representative MCP operations in this environment include:

- `dokploy-mcp_project-all`
- `dokploy-mcp_project-one`
- `dokploy-mcp_project-create`
- `dokploy-mcp_application-create`
- `dokploy-mcp_application-one`
- `dokploy-mcp_application-saveGithubProvider`
- `dokploy-mcp_application-saveGitProvider`
- `dokploy-mcp_application-saveDockerProvider`
- `dokploy-mcp_application-saveBuildType`
- `dokploy-mcp_application-saveEnvironment`
- `dokploy-mcp_application-deploy`
- `dokploy-mcp_application-redeploy`
- `dokploy-mcp_application-readAppMonitoring`

### Database workflow

Common sequence:

1. create
2. deploy
3. optionally expose an external port only if required
4. wire variables into the consuming application

Representative operations:

- `dokploy-mcp_postgres-create`
- `dokploy-mcp_postgres-deploy`
- `dokploy-mcp_postgres-saveEnvironment`
- `dokploy-mcp_postgres-saveExternalPort`
- `dokploy-mcp_mysql-create`
- `dokploy-mcp_mysql-deploy`

### Domain workflow

## Application behavior from Dokploy docs

- Applications expose logs, monitoring, deployments, variables, domains, and advanced settings.
- Domain configuration for applications is backed by Traefik file-provider config and can hot-reload without application redeploy.
- Advanced settings cover replicas, resources, volumes, ports, Traefik, and runtime commands.

## Docker Compose behavior from Dokploy docs

- Dokploy supports both normal Compose and Stack-oriented deployments.
- Stack mode is for Swarm orchestration and not every Compose feature is available there.
- UI variables become a `.env` file beside the compose file, but are **not automatically injected into containers** unless:
  - you use `env_file`, or
  - you explicitly reference `${VAR_NAME}` in the compose file.
- For private registries with Docker Stack and replicas, `--with-registry-auth` may be required so worker nodes can pull images.

## Variables behavior from Dokploy docs

Three scopes matter:

- **project-level** shared variables
- **environment-level** variables for environment-specific values
- **service-level** variables for overrides or service-specific config

Useful pattern:

- define shared secrets and connection strings once when possible
- reference them from dependent services rather than duplicating them

## Domains behavior from Dokploy docs

Applications and Compose stacks are managed differently:

- **Applications** use Traefik file-provider config generated by Dokploy.
- **Compose** uses Traefik Docker labels and requires redeploy for changes to take effect.

Important domain concepts:

- host
- path
- internal path
- strip path
- container port for internal Traefik routing
- HTTPS certificate mode

Path rewriting is powerful but easy to misuse; apps that generate absolute URLs may break if path rewriting is inconsistent with app configuration.

## Storage and persistence choices

Dokploy docs recommend:

- `../files/...` bind mounts for simple persistence and direct host access
- named volumes for Docker-managed persistence and Dokploy backup support

Good rule of thumb:

- if backup/restore matters, prefer named volumes
- if direct file editing on host matters more, bind mounts may be appropriate

## Verification checklist

After any meaningful change, verify at least the items that match the task:

- target resource exists and has the expected configuration
- deployment status progressed to healthy/running/done
- logs show successful startup rather than silent crash loops
- domain resolves to the right route and returns expected response
- data persists after redeploy if persistence was part of the task
- database is reachable from the intended app, not just “running” in isolation
